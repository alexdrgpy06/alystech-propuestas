# Implementation Notes: Refactor React+TS de la propuesta Araucanos

## Decisiones tomadas durante la implementación

- **Delegar la transcripción de contenido a un subagente** (`general-purpose`, ver historial de la sesión para el prompt completo) en vez de transcribir ~1600 líneas manualmente: el prompt incluía las interfaces TS ya cerradas y reglas de mapeo verificadas a mano (badge tone, feature status, cost category, matrix shape) para que la transcripción fuera mecánica y determinística, no interpretativa. Resultado verificado con typecheck + spot-check de 5 precios contra el HTML fuente — todos coincidieron.
- **`srv` y `net` son grupos (`GroupId`) separados pero comparten `stepIndex: 2`** en el wizard — en el HTML original están en el mismo `<section class="wstep" data-step="2">`. `net.inSimple` reutiliza el texto de `srv` (no hay stepcard propio para red en el original).
- **El paso 5 ("Tu selección") del original es dinámico**, no un Canvas matricial estático de 6 cuadrantes como el otro HTML de referencia (`canvas_plan_wizard_1.html`). El sitio en producción ya simplificó esto a: total + lista de ítems seleccionados + tabla de alternativas colapsable + 4 botones. Se replicó ese patrón simplificado en `CanvasStep.tsx`, no el patrón de 6 cuadrantes del otro archivo de referencia (que nunca se llegó a desplegar).
- **`ComparisonMatrix`, `GroupTechDetail` y `OnboardingContent` se extendieron varias veces** sobre la marcha porque el contenido real (especialmente el techdetail de Auditoría, que tiene narrativa + tabla comparativa bad/good + caja "cómo funciona paso a paso") no entraba en el modelo inicial más simple. Los tipos son ahora la fuente de verdad — cualquier componente nuevo debe leer `src/types/proposal.ts` antes de asumir una forma de dato.
- **El PDF se enriquece de forma aditiva**: `SelectionSummary` (en `types/proposal.ts`) agrega `description?` y `costBreakdown?` opcionales sobre la forma mínima que ya esperaba `server.js` (`{group, code, name, price, id}`). Esto significa que el payload que hoy arma `lib/api.ts` (`buildSelectionSummaries`) es compatible con el `/api/pdf` viejo sin romperlo — la reescritura del handler en `server.js` (tarea 7, pendiente) es la que debe empezar a usar esos campos nuevos.

## Gotchas / Edge Cases

1. **CRÍTICO — `npm run build` en `frontend/` borra `public/araucanos/index.html`.** `vite.config.ts` tiene `outDir: '../public/araucanos'` con `emptyOutDir: true`, y esa carpeta era el único lugar donde vivía el HTML original con todo el contenido de negocio. Esto ya pasó una vez en esta sesión (se corrió `build` antes de extraer el contenido) y se recuperó con `git show HEAD:public/araucanos/index.html`. **Mitigación ya aplicada:** existe una copia de seguridad permanente en `docs/reference/original-index.html`, fuera de `public/`, a salvo de futuros builds. **No borrar ese archivo** hasta que se verifique que absolutamente todo el contenido fue portado (incluyendo cualquier detalle menor que se pueda haber pasado por alto en `about.ts`, `terms.ts`, etc.). Si en algún momento hace falta releer el original, usar ese archivo, no `public/araucanos/index.html` (que ahora es el build de React).
2. **Tailwind v4, no v3.** No existe `tailwind.config.js`/`.ts`. Los tokens de color/fuente están en `@theme` dentro de `frontend/src/index.css`. Si algo busca un archivo de config JS de Tailwind, no lo va a encontrar — es esperado.
3. **`baseUrl` en `tsconfig.app.json` está deprecado en esta versión de TS** (~6.0.2, entorno de punta) — se removió y el alias `@/*` se resuelve solo con `paths` (sin `baseUrl`) más el alias espejado en `vite.config.ts` (`resolve.alias`). Si TS vuelve a quejarse de `paths` sin `baseUrl`, es una regresión a investigar, no algo para "arreglar" agregando `baseUrl` de nuevo sin revisar el warning de deprecación.
4. **El símbolo `JSX.Element` se usa en `GlanceGrid.tsx`** (un `Record<string, JSX.Element>` para el mapa de íconos) sin import explícito — depende de que el namespace `JSX` esté disponible globalmente vía `@types/react`. Si el typecheck falla ahí, puede hacer falta `import type { JSX } from 'react'` explícito (cambio menor, no estructural).
5. **`public/araucanos/` ahora está en `.gitignore`**, pero el `index.html` viejo sigue trackeado en git (nunca se hizo `git rm --cached`). Cuando se haga el primer commit de esta migración, `git status` va a mostrar el archivo como modificado (por el build nuevo) — decidir ahí si se deja de trackear explícitamente o se commitea el build output una vez (no recomendado, pero no es indispensable resolverlo ahora).
6. **No se tocó `server.js` todavía** salvo que la spec ya pre-aprobó reescribir específicamente el handler de `/api/pdf` (no otros endpoints). No hace falta volver a pedir permiso para ese cambio puntual.
7. **El "Overview"/resumen ejecutivo del paso 0 y el "Escalable y mejorable" banner después del onboarding no estaban en el alcance original del prompt de extracción** dado al subagente — se detectaron como huecos después (grep de "El problema"/"Una sola infraestructura" sin resultados) y se completaron a mano en `data/overview.ts` y como campo `scaleBanner` en `data/onboarding.ts`. Si algo más parece faltar, comparar contra `docs/reference/original-index.html` con grep antes de asumir que está cubierto.

## Deviations from Plan

- El plan original (`docs/spec-react-refactor.md` → Implementation Plan) preveía fases A-G más o menos secuenciales. En la práctica, la fase B (modelo de datos) se hizo con ayuda de un subagente en paralelo mientras se armaba el scaffolding, y la fase D (componentes) arrancó por los átomos reutilizables (Badge, FeatureList, CostBreakdownTable, Modal) antes que por el layout de más alto nivel — es más fácil verificar tipos de abajo hacia arriba. No afecta el resultado final, solo el orden de construcción.

## Performance / Security Notes

- Nada crítico identificado todavía. Pendiente de revisar cuando se integre `App.tsx`: el `AnimatePresence`/`motion` de Framer Motion en `Modal.tsx` ya respeta `prefers-reduced-motion` a nivel global vía el CSS en `index.css` (`@media (prefers-reduced-motion: reduce)` fuerza `animation-duration`/`transition-duration` a ~0), pero no se verificó explícitamente que Framer Motion (que anima con JS, no solo CSS) respete esa media query — Framer Motion tiene su propio hook `useReducedMotion()` que **no se está usando todavía**. Revisar antes de dar la tarea 6 (animaciones) por terminada.
